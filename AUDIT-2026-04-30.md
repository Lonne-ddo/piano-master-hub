## Issues détectées

### [P1] Auth — pas de revalidation `isAdminEmail` au verify
**Catégorie** : Sécurité — defense-in-depth
**Fichier** : `functions/api/auth/verify.js` L40-58
**Description** : Si Estelon retire un email de `ADMIN_EMAILS` après que ce compte a déjà
demandé un magic link, le verify créera quand même une session admin (basée uniquement sur
le contenu `magic_link.is_admin` du KV). Fenêtre 15 min max (TTL magic_link), mais possible.
**Impact** : Offboarding admin partiellement effectif pendant 15 min après changement de
config ADMIN_EMAILS. Pas exploitable par un tiers, mais incohérent avec l'intention.
**Effort fix** : XS
**Statut** : 🔧 AUTO-FIX
**Fix** : importer `isAdminEmail`, recheck `data.email` contre `env.ADMIN_EMAILS` avant de
créer la session admin. Si plus admin → redirect `?error=expired`.

### [P1] Auth — session admin reste valide 90j même après retrait de ADMIN_EMAILS
**Catégorie** : Sécurité — defense-in-depth
**Fichier** : `functions/api/eleves/[slug]/public.js` L36-50
**Description** : Une fois la session admin créée (cookie 90j + KV `session:<token>` 90j),
le bypass `if (!session.is_admin && session.slug !== slug)` reste actif jusqu'à expiration
du cookie ou logout explicite, **même si l'email est retiré de `ADMIN_EMAILS`**.
**Impact** : Offboarding admin nécessite logout explicite ou attente 90j. Pas de mécanisme
côté serveur pour invalider. Pour MVP avec 1 admin (Estelon lui-même qui contrôle CF env vars
ET son cookie), pas critique. Mais pour multi-admin futur, c'est un trou.
**Effort fix** : XS
**Statut** : 🔧 AUTO-FIX
**Fix** : revalider `isAdminEmail(session.email, env)` à chaque accès `/api/eleves/{slug}/public`
si `session.is_admin === true`. Si plus admin → 401 (force re-login). Coût : 1 split CSV par
GET, négligeable.

### [P2] Auth — timing attack anti-énumération
**Catégorie** : Sécurité — fuite d'info
**Fichier** : `functions/api/auth/request-link.js` L60-72
**Description** : Email reconnu (élève ou admin) → KV put + Resend API call (~300-1500ms).
Email inconnu → silent return (~5-10ms). Différence mesurable côté attaquant pour énumérer.
**Impact** : Attaquant patient peut deviner les emails enregistrés. Bénéfice limité (4 élèves
+ 1 admin total).
**Effort fix** : S
**Statut** : 📋 BACKLOG
**Proposition** : 3 options — (a) délai constant `await sleep(1000)` minimum dans tous les
chemins, (b) faire le Resend send en async fire-and-forget en parallèle de la réponse,
(c) accepter le trou pour MVP. Recommandation MVP : (c) avec note dans CLAUDE.md.

### [P2] Bibli — pas de reset de transposition/tempo au nouveau morceau
**Catégorie** : UX
**Fichier** : `bibli.js` L317-340 (searchSong)
**Description** : `state.transposition` et `state.tempo` persistent entre recherches via
localStorage. Si l'élève a transposé "La Bohème" de +3, puis cherche "Imagine", il garde
+3 → première lecture désaccordée. Le tempo a une logique sentinel (state.tempo === 100 →
adopt LLM bpm), donc partiellement OK, mais transpose pas reset.
**Impact** : Première écoute du nouveau morceau dans une mauvaise tonalité. Bouton transpose
+/- visible permet correction.
**Effort fix** : XS
**Statut** : 📋 BACKLOG
**Proposition** : reset `state.transposition = 0` au début de `searchSong()`. Pour le tempo,
toujours adopter le bpm LLM au nouveau résultat (overwrite plutôt que sentinel).

### [P2] Bibli — LLM prompt injection limitée
